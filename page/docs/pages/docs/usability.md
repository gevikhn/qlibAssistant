# qlibAssistant 可用性分析

## 结论摘要

从当前仓库结构、文档内容和自动化流程来看，**qlibAssistant 对“只想查看每日选股结果的用户”已经具备较好的可用性**，因为页面入口清晰、结果更新自动化、复盘数据可直接查看。  
但对“希望本地复现、二次开发或长期维护项目的用户”而言，当前可用性仍然是**中等水平**：核心能力完整，但环境准备、依赖说明、测试入口和故障自助排查信息还不够集中。

## 适用用户分层

### 1. 结果查看型用户：可用性较好

这类用户只关心两个问题：**今天有什么结果**、**历史上准不准**。

当前项目已经提供了较顺畅的使用路径：

- README 首页直接给出“每日预测结果”和“复盘统计”链接
- 页面端按日期展示结果，适合快速浏览
- 帮助页解释了 `avg_score`、`pos_ratio`、`xxx_ret`、`xxx_filter_ret` 等关键字段
- 项目强调“仅供参考，不构成投资建议”，降低误用风险

因此，对于“打开网页看结果”的轻量用户，学习成本较低，入口也比较直接。

### 2. 本地运行型用户：可用性一般

这类用户需要自己安装依赖、同步数据、解压模型并执行预测。

当前优点：

- README 已经给出最短运行命令
- `roll.py` 采用命令行子模块设计，入口相对统一
- GitHub Actions 工作流帮助用户理解自动化执行顺序

当前阻力：

- 依赖较重，且 `requirements.txt` 中主要列出了运行依赖，没有单独说明测试依赖或版本兼容范围
- 数据、模型、页面三部分分散在不同目录，新用户需要先理解仓库结构
- 部分说明分布在 README、帮助页和 workflow 中，信息入口不够收敛
- 项目对外部数据源、GitHub Actions 海外环境限制有说明，但缺少更系统的排障步骤

综合来看，本地运行并非不可行，但首次上手仍有一定门槛。

### 3. 维护/二开型用户：可用性中等偏弱

对于想修改策略、替换模型或扩展页面的开发者，当前仓库已经提供了基础模块划分：

- `roll/`：数据、训练、模型分析主逻辑
- `script/`：批处理与辅助脚本
- `page/`：结果页面生成与展示
- `.github/workflows/`：自动化调度流程

不过，从维护体验角度看，还存在一些影响效率的点：

- 测试、构建、部署信息分别位于多个位置，缺少“开发者快速开始”集中说明
- 自动生成页面与手写页面共存，需要先理解生成规则再修改
- CI 依赖外部数据与定时任务，问题定位成本相对较高

## 当前可用性的主要优势

### 1. 用户目标明确

项目核心目标非常聚焦：**自动生成并展示每日 CSI300 选股结果，并提供复盘验证**。  
目标单一意味着页面和文档都能围绕“看结果”和“查历史表现”来设计，这对用户理解非常有帮助。

### 2. “看结果”路径短

从 README 到网页结果页，路径很短，适合非开发用户直接消费成果。  
对于多数开源项目来说，能把“先跑起来”变成“先看到结果”，本身就是明显的可用性优势。

### 3. 自动化流程降低了日常使用成本

项目通过 GitHub Actions 自动完成训练、分析、复盘与页面部署，减少了人工操作。  
这让用户不需要每天手动执行整套量化流程，可用性明显优于纯脚本型项目。

### 4. 关键术语已有解释

帮助页已经对分数、文件类型和常见问题做了说明，这降低了金融/量化领域术语带来的理解成本。

## 当前可用性的主要短板

### 1. 首次搭建门槛偏高

项目依赖量化框架、数据源、模型文件和页面生成工具，对完全新手并不友好。  
虽然 README 提供了命令示例，但“每一步为什么要做、失败后怎么排查”还可以更明确。

### 2. 说明信息分散

目前 README、帮助页、页面说明、workflow 配置分别承担了不同的信息职责。  
这种结构对熟悉仓库的人没有问题，但对新用户来说，容易出现“知道有文档，但不知道先看哪一页”的情况。

### 3. 结果消费优先，开发体验次之

项目更偏向“把结果稳定生产出来并展示”，这对最终结果用户是加分项；  
但对开发者而言，测试入口、依赖边界、目录职责、页面生成规则还不够集中，导致认知切换成本较高。

### 4. 故障处理知识还不够显式

仓库已经提到外部数据接口、海外网络环境、模型筛选为空等问题，但这些内容主要散落在说明文字中。  
如果能进一步沉淀成更结构化的排障清单，可用性会明显提升。

## 综合判断

如果以 5 分制做一个主观评价：

| 维度 | 评价 | 说明 |
|------|------|------|
| 查看结果 | 4.5 / 5 | 入口直接、页面明确、复盘可查 |
| 本地使用 | 3 / 5 | 能跑，但依赖与准备步骤偏重 |
| 开发维护 | 3 / 5 | 结构清楚，但开发说明仍可继续集中化 |
| 故障自助排查 | 2.5 / 5 | 已有提示，但还不够系统 |

**总体可用性判断：3.5 / 5。**  
项目已经很好地服务了“结果消费”这一核心场景，但如果希望让更多用户稳定完成本地复现、二次开发和问题自查，文档组织和开发者引导仍有提升空间。

## 建议的优化优先级

若后续继续提升可用性，建议优先考虑以下顺序：

1. **补一份开发者快速开始文档**：把环境、依赖、测试、页面构建、常见错误集中到一页
2. **补一份排障清单**：明确数据更新失败、模型为空、页面未生成等典型问题的处理方式
3. **明确测试依赖与执行方式**：降低维护者验证改动的成本
4. **在页面端补充“我是新用户该看哪里”导航**：继续缩短首次理解路径

以上优化不一定需要重构代码，很多只靠文档整合就能明显提升项目可用性。
